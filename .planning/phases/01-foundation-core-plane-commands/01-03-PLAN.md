1. **server/dialog.go** — Dialog construction:

   `openCreateTaskDialog(p *Plugin, triggerID, channelID, userID string) error`:
   - Construct `model.OpenDialogRequest` with:
     - CallbackId: "create_task"
     - Title: "Create Task in Plane"
     - URL: `fmt.Sprintf("/plugins/%s/api/v1/dialog/create-task", manifest.Id)`
     - SubmitLabel: "Create Task"
     - Elements:
       a. **Title** (text, required): DisplayName "Title", Name "title", SubType "text", MinLength 1, MaxLength 255
       b. **Description** (textarea, optional): DisplayName "Description", Name "description"
       c. **Project** (select, required, dynamic): DisplayName "Project", Name "project_id", DataSource "dynamic". The dynamic select URL is NOT set here — Mattermost handles it via the plugin's `/api/v1/select/projects` endpoint registered as a separate lookup mechanism. Actually: for Mattermost dialogs, `DataSource: ""` with static options OR `DataSource: "dynamic"` but the data source for dynamic selects in dialogs is NOT a URL — it's handled differently than post actions. RESEARCH NOTE: Mattermost interactive dialog dynamic selects DO NOT use data_source_url. Instead, use static options pre-populated from Plane API at dialog-open time. Fetch projects list when opening dialog and populate as static options.
       d. **Priority** (select, optional): DisplayName "Priority", Name "priority", static options: None/Low/Medium/High/Urgent. Default "none".
       e. **Assignee** (select, optional, pre-populated): DisplayName "Assignee", Name "assignee_id". Fetch members from first/selected project at dialog-open time. Default: current user's Plane ID.
       f. **Labels** (text, optional): DisplayName "Labels (comma-separated)", Name "labels". Since multi-select in dialogs is uncertain (Research Open Question #3), use a text field where user types comma-separated label names. The submission handler will resolve names to IDs.

   IMPORTANT: Since interactive dialogs don't support true dynamic selects, pre-populate project, assignee, and label options at dialog-open time by calling Plane API (cached). If Plane has many projects, the first project's members and labels are loaded initially.

   Pre-population flow in `openCreateTaskDialog`:
   - Fetch `p.planeClient.ListProjects()` -> populate project select options
   - If projects exist, fetch members and labels for the first project
   - Set assignee default to current user's Plane ID (from store mapping)

2. **Update server/command_handlers.go** — Replace handlePlaneCreate stub:

   `handlePlaneCreate(p *Plugin, c *plugin.Context, args *model.CommandArgs, subArgs []string) *model.CommandResponse`:
   a. Call `requirePlaneConnection()` — if not connected, return (guard already sends ephemeral)
   b. Check if Plane client is configured — if not, ephemeral error
   c. **Quick inline mode**: If subArgs has content after "create"/"c" (i.e., the user typed `/task plane create Fix the login bug` or `/task p c "Fix the login bug"`), extract the title text. Strip surrounding quotes if present. Then:
      - Get user's Plane mapping from store
      - Get list of projects. If only 1 project, use it. If multiple, use the first one (or error asking user to use the full dialog).
      - Call `p.planeClient.CreateWorkItem(projectID, &CreateWorkItemRequest{Name: title, Assignees: []string{mapping.PlaneUserID}, Priority: "none"})`
      - Send ephemeral confirmation (see Task 2 for format)
      - Return empty CommandResponse
   d. **Dialog mode**: If no arguments after "create", open interactive dialog via `openCreateTaskDialog()`. The dialog needs `args.TriggerId` — use it immediately (trigger IDs expire quickly). Return empty CommandResponse.

3. **Update server/api.go** — Implement handleCreateTaskDialog (replace stub):

   `handleCreateTaskDialog(w http.ResponseWriter, r *http.Request)`:
   a. Parse `model.SubmitDialogRequest` from request body
   b. Extract fields from `Submission` map: title (string), description (string), project_id (string), priority (string), assignee_id (string), labels (string, comma-separated)
   c. Validate: title must not be empty. If empty, return dialog error response: `{"errors": {"title": "Title is required"}}`
   d. Resolve labels: if labels text is non-empty, split by comma, trim spaces. Fetch `p.planeClient.ListProjectLabels(projectID)` and match by name (case-insensitive). Collect matching label IDs. Log warning for unmatched names.
   e. Build `CreateWorkItemRequest` with all fields
   f. Call `p.planeClient.CreateWorkItem(projectID, req)`
   g. On success: get project name from cached projects list. Build confirmation message. Send ephemeral to user.
   h. On error: send ephemeral error to user with actionable message
   i. Return 200 with empty JSON to dismiss dialog

   i. Send as ephemeral message

   a. Call `requirePlaneConnection()` — guard
   b. Determine project: check subArgs for project name/identifier. If provided, find matching project from `ListProjects()` (match by name or identifier, case-insensitive). If not provided and there's only one project, use it. If multiple projects and none specified: ephemeral listing available projects asking user to specify: "Which project? Available: {list}. Usage: `/task plane status {project}`"
   c. Fetch all states for the project: `p.planeClient.ListProjectStates(projectID)`
   d. Fetch work items for the project (all, no assignee filter — up to 100 via per_page): This uses `ListProjectWorkItems(projectID string) ([]WorkItem, error)` in `server/plane/work_items.go` that fetches WITHOUT assignee filter: `GET /projects/{pid}/work-items/?per_page=100&expand=state_detail`.
   e. Group work items by state group (backlog, unstarted, started, completed, cancelled)
   f. Calculate counts and totals
   g. Format as specified in CONTEXT.md:
      ```
      **Project: {projectName}** ({identifier})

      | Status | Count |
      |--------|-------|
      | :white_circle: Open | {count} |
      | :large_blue_circle: In Progress | {count} |
      | :white_check_mark: Done | {count} |

      **Progress:** [{progressBar}] {percent}%
      **Total:** {total} work items

      [Open in Plane]({projectURL})
      ```
      Progress bar: Map "started" group to "In Progress", combine "backlog"+"unstarted" as "Open", "completed" as "Done". Bar is ASCII: `[========------]` style using block characters.
   h. Send as ephemeral message
