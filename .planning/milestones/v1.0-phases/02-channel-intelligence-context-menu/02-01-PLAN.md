<objective>
Implement channel-project binding and make all existing commands binding-aware.

Purpose: Channels become project workspaces. Binding a channel to a Plane project means every command in that channel automatically knows which project to use, eliminating the need to specify it each time. This is the foundational capability that makes the context menu (Plan 03) project-aware.

Output: Working `/task plane link` and `/task plane unlink` commands, plus all existing handlers (create, mine, status) automatically using the bound project when executed in a linked channel.
</objective>

**2. `server/command_handlers_binding.go`** -- New file with two handlers:

`handlePlaneLink(p, c, args, subArgs)`:
- Call `requirePlaneConnection` (user must be connected to Plane).
- If no subArgs, list available projects and respond ephemeral with usage: `/task plane link <project name>`.
- Join subArgs as project query, call `p.planeClient.ListProjects()`, find match via `findProjectByNameOrID`.
- If not found, respond ephemeral with available projects.
- If found, create `ChannelProjectBinding{ProjectID, ProjectName, BoundBy: args.UserId, BoundAt: time.Now().Unix()}`.
- Call `p.store.SaveChannelBinding(args.ChannelId, &binding)`.
- Post a **visible** message to the channel (not ephemeral): `p.API.CreatePost(&model.Post{UserId: p.botUserID, ChannelId: args.ChannelId, Message: fmt.Sprintf(":link: Canal vinculado al proyecto **%s**", projectName)})`.
- Return empty `&model.CommandResponse{}`.

`handlePlaneUnlink(p, c, args, subArgs)`:
- Call `requirePlaneConnection`.
- Check `p.store.GetChannelBinding(args.ChannelId)`. If nil, respond ephemeral "Este canal no esta vinculado a ningun proyecto. Usa `/task plane link <proyecto>` para vincularlo."
- If bound, call `p.store.DeleteChannelBinding(args.ChannelId)`.
- Post visible message: `p.API.CreatePost` with ":broken_chain: Canal desvinculado del proyecto **{ProjectName}**".
- Return empty response.

**1. Modify `server/dialog.go`** -- Refactor `openCreateTaskDialog` to accept optional context:

Create a new function `openCreateTaskDialogWithContext` with signature:
```go
func openCreateTaskDialogWithContext(p *Plugin, triggerID, channelID, userID, preTitle, preDescription string, binding *store.ChannelProjectBinding, sourcePostID string) error
```

This function is the new implementation. It does everything `openCreateTaskDialog` does, plus:
- If `preTitle != ""`, set the title element's Default to `preTitle`.
- If `preDescription != ""`, set the description element's Default to `preDescription`.
- If `binding != nil`, set the project_id element's Default to `binding.ProjectID` instead of `projects[0].ID`.
- If `sourcePostID != ""`, add a hidden element `source_post_id` (type "text", Optional: true, Default: sourcePostID) so the dialog submission handler knows which post to react to. Use `SubType: "text"` and set it as a hidden field -- since Mattermost dialogs don't have a true hidden type, use a workaround: add it as a regular text field but prefix DisplayName with empty string and set it to optional. Alternative: pass it via the callback URL as a query parameter: `URL: fmt.Sprintf("/plugins/%s/api/v1/dialog/create-task?source_post_id=%s", manifestID, sourcePostID)`.

Refactor existing `openCreateTaskDialog` to call the new function with empty defaults:
```go
func openCreateTaskDialog(p *Plugin, triggerID, channelID, userID string) error {
    binding, _ := p.store.GetChannelBinding(channelID)
    return openCreateTaskDialogWithContext(p, triggerID, channelID, userID, "", "", binding, "")
}
```

This way, even the existing `/task plane create` dialog mode now checks for binding and pre-selects the bound project.
