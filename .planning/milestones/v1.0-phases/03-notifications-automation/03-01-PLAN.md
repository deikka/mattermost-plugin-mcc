**server/webhook_plane.go** -- Replace the stub `handlePlaneWebhook` with full implementation:

1. `handlePlaneWebhook(w, r)`:
   - Read body via `io.ReadAll(r.Body)`
   - Verify HMAC via `verifyWebhookSignature(body, r.Header.Get("X-Plane-Signature"))` -- 403 on failure
   - Deduplicate via `isWebhookDuplicate(r.Header.Get("X-Plane-Delivery"))` -- 200 on duplicate
   - Call `markWebhookProcessed(deliveryID)`
   - Parse `PlaneWebhookEvent` from body
   - Switch on `event.Event`:
     - `"issue"` + `action=="updated"`: call `p.handleIssueWebhook(&event)`
     - `"issue_comment"` + `action=="created"`: call `p.handleIssueCommentWebhook(&event)`
   - Return 200 with `{"status": "ok"}`

2. `handleIssueWebhook(event *PlaneWebhookEvent)`:
   - Parse `event.Data` into `WebhookIssueData`
   - Check `plugin_action_{workItemID}` KV key -- if present, skip (self-notification suppression). Use `p.API.KVGet("plugin_action_" + issueData.ID)`
   - Look up bound channels: `p.store.GetProjectChannels(issueData.Project)`
   - For each channel:
     - Check `p.store.GetNotificationConfig(channelID)` -- skip if Enabled==false
     - Detect change type:
       - **State change:** Read cached state from `work_item_state_{workItemID}` KV key. If cached state differs from current `issueData.State.Group`, it's a state change. Build card via `buildStateChangeAttachment`. Cache the new state.
       - **Assignee change detection:** Since we don't have "previous" assignees from the webhook, and per research Pitfall 1 the webhook only sends current state, we take a pragmatic approach: always post a "current state" card on issue update that includes state info. To detect assignee changes specifically, cache the last known assignee list hash. If no previous cache exists, this is the first event we see -- cache current state but don't notify (avoids spurious "change" notifications on first webhook for existing items).
     - Post via `p.API.CreatePost` with bot user ID, channel ID, and SlackAttachment
   - After posting, update state cache: `work_item_state_{workItemID}` with TTL 7 days (604800 seconds)

   **Simplification per user decisions:** "Todos los cambios visibles para todos" + "cada cambio = una notificacion". On every issue update webhook:
   - Always cache current state
   - If we have a previous cached state AND it differs from current: post state change card
   - The card always shows current assignees regardless (enriches the notification)

3. `handleIssueCommentWebhook(event *PlaneWebhookEvent)`:
   - Parse `event.Data` into `WebhookCommentData`
   - The comment payload may not include the parent issue project. Per research Open Question #1, parse defensively. If `IssueProject` field is empty, try to extract project from the webhook's `Data` or log a warning and skip.
   - For comment events, we need the project ID to route. The comment webhook data includes an `issue` field (the work item ID). We need the project to find bound channels. Strategy: since the webhook top-level has `workspace_id`, and we need project_id, check if the comment data includes a `project` field. If not, we'll need to iterate all project_channels_ to find which project owns this issue. For simplicity and per research recommendation, use `json.RawMessage` parsing and extract `project` if present. If not present, skip notification with a log warning.
   - Route to bound channels same as issue events
   - Build comment card with truncated text

4. Notification card builders (new functions, NOT the same as link unfurl's `buildWorkItemAttachment`):
   - `buildStateChangeAttachment(taskName, oldState, newState, oldStateGroup, newStateGroup, actorName, taskURL string) *model.SlackAttachment`:
     - Title: `"{emoji} Estado cambiado: {taskName}"` where emoji = `stateGroupEmoji(newStateGroup)`
     - TitleLink: taskURL
     - Fields: `[{Title: "Cambio", Value: "{oldState} -> {newState}", Short: true}, {Title: "Por", Value: actorName, Short: true}]`
     - Color: "#3f76ff", Footer: "Plane"
     - If oldState is empty (no cache), show `"-> {newState}"` instead of `"{oldState} -> {newState}"`
   - `buildAssigneeChangeAttachment(taskName string, assignees []WebhookAssignee, taskURL string) *model.SlackAttachment`:
     - Title: `"Asignacion cambiada: {taskName}"`
     - Field: assignee names joined with ", "
     - Color: "#3f76ff", Footer: "Plane"
   - `buildCommentAttachment(taskName, commentText, actorName, taskURL string) *model.SlackAttachment`:
     - Title: `"Nuevo comentario: {taskName}"`
     - TitleLink: taskURL
     - Text: truncated comment (~200 chars). Strip HTML tags from `comment_html` using a simple regex `<[^>]*>` replacement before truncating. Append "..." if truncated.
     - Fields: `[{Title: "Por", Value: actorName, Short: true}]`
     - Color: "#3f76ff", Footer: "Plane"

5. Helper: `buildTaskURL(planeURL, workspace, projectIdentifier string, sequenceID int) string` -- builds `{planeURL}/{workspace}/browse/{identifier}-{sequenceID}`. If projectIdentifier is unknown (webhook doesn't include it), build a direct URL: `{planeURL}/{workspace}/projects/{projectID}/work-items/{workItemID}`.

6. Helper: `stripHTMLTags(s string) string` -- regex `<[^>]*>` -> "" for stripping HTML from comment_html.

7. Helper: `truncateText(s string, maxLen int) string` -- truncate to maxLen chars, append "..." if truncated.
